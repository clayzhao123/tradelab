# V1 Domain Glossary

## Purpose

This glossary freezes domain terms for database naming, API contracts, websocket events, and frontend entities.
Do not invent synonyms in code once a canonical term exists.

## Core Terms

### run
- Definition: A single executable lifecycle instance of strategy operation in live paper-trading mode.
- Lifecycle: `pending` -> `running` -> `stopped` or `completed` or `failed`.
- Constraints: At most one active (`running`) run at a time in V1.

### strategy
- Definition: Persisted configuration set that defines entry/exit/risk parameters for a run.
- Notes: Strategy is configuration; run is execution instance.

### order
- Definition: An instruction to buy or sell a symbol, created by user or strategy execution.
- Core fields: side, type, quantity, requested price (if applicable), status, timestamps.
- Typical statuses: `new`, `open`, `partial`, `filled`, `cancelled`, `rejected`, `expired`.

### fill
- Definition: A partial or complete execution fragment of an order.
- Relationship: Many fills may belong to one order.
- Effect: Fills mutate position and account exposure/cash state.

### position
- Definition: Current net holdings and cost basis for a symbol derived from fills.
- Notes: Not user-entered directly; computed/maintained by execution lifecycle.

### account snapshot
- Definition: Point-in-time account state including cash, equity, unrealized and realized PnL, exposures.
- Usage: Render dashboard and persist historical run analysis.

### risk rule
- Definition: Configurable constraint used by risk engine to validate actions and runtime state.
- Examples: max symbol exposure, gross exposure, cash floor, max drawdown.

### risk event
- Definition: Recorded event when a risk rule is evaluated as violated or near-violated.
- Usage: Both realtime alerting and historical audit.

### quote
- Definition: Latest market price tuple for symbol (bid/ask/last/timestamp as available).

### kline
- Definition: Time-bucketed OHLCV market data for charting and indicators.

### scanner result
- Definition: Ranked output row generated by scan logic over candidate symbols at a timestamp.

## Naming Rules (API + Events + Data)

- Use singular nouns for entity endpoints roots where already defined in plan, e.g. `/orders`, `/fills`, `/runs`.
- Use explicit action verbs only for non-CRUD actions:
  - `/runs/start`
  - `/runs/:id/stop`
  - `/scan/run`
- Websocket event names use `<entity>.<action>`:
  - `order.updated`
  - `fill.created`
  - `run.updated`
  - `account.updated`
  - `risk.triggered`
  - `snapshot` (reserved for hydration payload)
- Avoid aliases:
  - Use `run` not `session` in backend contract names.
  - Use `fill` not `trade` for execution fragments.
  - Use `risk event` not `alert` for persisted violation records.

## Contract Consistency Checklist

Before adding any API or event:
- Does name match glossary canonical term?
- Is lifecycle state modeled using existing status vocabulary?
- Is term reused consistently in route, DTO, DB schema, and frontend entity mapper?
